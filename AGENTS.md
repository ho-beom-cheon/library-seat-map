# AGENTS.md

이 문서는 `library-seat-map` 저장소에서 Codex 같은 자동화 에이전트가 작업할 때 따라야 하는 유지보수 규칙입니다.

## 기본 원칙

- 모든 작업은 이슈 기반으로 진행합니다.
- 구현 전에 관련 이슈를 확인합니다.
- 관련 이슈가 없으면 구현하지 말고 이슈 초안을 먼저 제안합니다.
- 작업은 `codex/<short-task-name>` 또는 `codex/issue-<number>-<short-task-name>` 형식의 전용 브랜치에서 진행합니다.
- 변경은 작은 단위로 나누고, PR 설명에는 변경 이유와 검증 내용을 명확히 씁니다.
- 작업 완료 시 릴리스 계획 문서에 반영할 후보 내용을 제안합니다.
- 의미 없는 커밋, 불필요한 파일 정리, 활동량만 늘리기 위한 변경은 하지 않습니다.
- 사용자 요청 없이 기존 변경사항을 되돌리지 않습니다.

## 작업 범위

- 애플리케이션 소스 변경은 실행 동작에 영향을 줄 수 있는 파일을 포함합니다.
- 문서, 이슈 템플릿, PR 템플릿, 릴리스 관리 문서는 운영 체계 변경으로 취급합니다.
- 요청 범위 밖의 파일은 수정하지 않습니다.
- 기존 변경이 보이면 되돌리지 말고, 현재 작업과 충돌하는 경우에만 사용자에게 확인합니다.
- 문서 전용 작업에서는 애플리케이션 소스, 빌드 설정, 의존성 파일을 변경하지 않습니다.

## 프로젝트 주의사항

- 공공데이터포털 서비스키와 지도 SDK 키는 환경변수 또는 배포 Secret으로만 관리합니다.
- 서비스키, 토큰, 내부 접속 정보, 개인 식별 정보는 문서, 코드, 로그에 포함하지 않습니다.
- 프론트엔드에서 공공데이터 API를 직접 호출하지 않습니다.
- 사용자 현재 위치 좌표는 저장하지 않고, 요청 처리와 추천 계산에만 사용합니다.
- 좌석 정보는 보장 표현이 아니라 최근 갱신 기준의 가능성 표현으로 작성합니다.
- "좌석 보장", "예약 가능", "무조건 앉을 수 있음" 같은 표현은 사용하지 않습니다.

## 권장 작업 흐름

1. 관련 이슈가 있는지 확인합니다.
2. 이슈가 없으면 문제, 제안, 완료 기준을 포함한 이슈 초안을 제안합니다.
3. 전용 브랜치를 생성하거나 현재 브랜치가 작업 목적에 맞는지 확인합니다.
4. 변경 범위를 작은 단위로 유지합니다.
5. 필요한 검증을 실행하고 결과를 기록합니다.
6. PR 설명을 작성합니다.
7. 릴리스 계획 문서에 반영할 항목을 제안합니다.

## 커밋과 PR

- 커밋 메시지는 영어 Conventional Commits 형식을 사용합니다.
- PR 제목과 본문은 한국어로 작성합니다.
- 한 커밋에는 한 가지 목적만 담습니다.
- 관련 이슈가 있으면 PR 본문에 `Closes #번호` 형식으로 연결합니다.
- 검증하지 못한 항목은 체크하지 않고, 미검증 사유를 적습니다.

## 검증 기준

문서 변경이면 다음을 확인합니다.

```bash
git diff --name-only
git diff --check
rg -n "AKIA|SECRET|PASSWORD=|TOKEN=|PRIVATE KEY|BEGIN RSA|REPO_PASS|REPO_USER|internal|고객사|내부정보" .
```

애플리케이션 소스 변경이 없어야 하는 작업에서는 다음을 별도로 확인합니다.

```bash
git diff --name-only -- src build.gradle settings.gradle package.json package-lock.json pnpm-lock.yaml yarn.lock
```
